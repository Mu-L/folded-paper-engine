UI and Cursors
UI Options, UI Cursor, and keeping UI visible with your camera

UI Options (menu entries)

  • UI Options give you named menu entries you can show on screen (e.g., Pause, Resume, Settings). The engine stores them globally so your UI scene can list what is available.
  • Create your own Godot UI scene and read the registered UI Options to populate buttons or toggles. This keeps menus designer-friendly while still reflecting the runtime state.

UI Cursor (crosshair and pointer)

  • Mark an object in Blender as UI Element if you want to turn it into a UI Cursor or UI Option. The flag simply tells the runtime the node can be used for UI—how it appears depends on which role you choose.
  • Place the UI cursor geometry where the player camera can see it and keep it in front of the camera. The engine does not automatically reposition UI Elements; you keep them visible by parenting to the camera or by animating them into view.
  • At runtime, the engine can show a simple crosshair and optionally unlock the mouse for menu navigation. Use a pause/menu trigger to toggle the cursor lock state.

Practical setup

  1. Create a small crosshair mesh or sprite in Blender, mark it as UI Element, and make it a child of your camera so it stays in view.
  2. For menus, build a Godot UI scene that reads UI Options and shows buttons. When you pause the game, unlock the mouse and show this scene; when you resume, lock the mouse again and hide the UI.
  3. If you need a custom cursor graphic, swap the UI Cursor object in your Godot scene with your own UI Element so the appearance matches your project.

Helpful reminders

  • UI Elements are not the same as Godot’s Control nodes. They are 3D objects you keep in front of a camera to act as cursors or options.
  • If you change cameras during a cutscene, be sure the UI Element is parented to the active camera or otherwise moved so it stays visible.
  • Use Deactivate Player Controls when showing menus so players do not move while navigating UI.
