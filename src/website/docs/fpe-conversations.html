Conversations
Speaker-tagged actors, dialogue data, and reply flow

Mark characters as Speakers in Blender

  1. Select the character or prop that should talk.
  2. In the Folded Paper Engine panel, open Speaker and check Speaker so the runtime knows this node can speak.
  3. If the Speaker should also play 3D audio, set the Speaker File and optional autoplay/loop settings described in the Audio guide.
  4. Export to GLB. The Speaker flag comes through to Godot so conversations and sounds can target the node by name.

Author the dialogue

  • Create a conversation resource in Godot with your comments and replies. Add the character names you want displayed to the player.
  • Assign that conversation asset to the Speaker node (or a parent) so the runtime can find it when a trigger fires.
  • Each reply can point to the next comment, allowing branching choices. Keep labels simple (“Greet”, “Ask for help”) so creators can follow the flow.

Start and advance a conversation

  1. Add a trigger near the NPC and give it a StartConversation command that points at the Speaker. You can also call the command from another event, like a frame event in a cutscene.
  2. When the command runs, a conversation instance is created and locked to the Speaker. Replies advance when the player chooses an option or when you call the reply helper from code.
  3. When there are no replies left, the conversation ends automatically and control returns to the player.

Friendly tips

  • Keep Speaker node names readable (“GuideNPC”) so your commands can target them without guesswork.
  • Pair conversations with a Deactivate Player Controls command if you want the player to stand still during dialogue, then re-enable controls afterward.
  • Reuse the Speaker Trigger command for simple voice lines that do not need branching replies.
